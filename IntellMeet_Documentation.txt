1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
What IntellMeet is:
IntellMeet is a comprehensive, full-stack video conferencing and real-time collaboration platform designed for professional and secure communication. Built using the MERN stack (MongoDB, Express.js, React.js, Node.js) alongside WebRTC and Socket.io, it offers an end-to-end solution for modern remote meetings.

Purpose of the project:
The primary purpose of IntellMeet is to provide a seamless, highly accessible, and intelligent meeting environment that doesn't just connect people visually and audibly, but also enhances productivity through AI-driven meeting summaries and action item generation.

Problems it solves:
- Dependency on expensive, enterprise-locked video conferencing tools.
- Difficulty in manually tracking meeting minutes and assigning action items.
- Lack of customizable, secure, and privacy-focused communication channels.

Main objective of the platform:
To deliver a robust, scalable, and responsive real-time communication platform where users can instantly create or join secure meeting rooms, share their audio/video streams, communicate via text, and obtain automated intelligent meeting summaries.

Real-world use case of the project:
- Telehealth/Medical consultations where a doctor and patient can securely connect.
- Remote enterprise team stand-ups and sprint planning.
- Online education and virtual classrooms.

Why this project is useful:
It packages essential remote collaboration features into an intuitive interface while introducing cutting-edge AI summarization, effectively reducing the administrative overhead of post-meeting reviews.

4. PROJECT ARCHITECTURE
--------------------------------------------------------------------------------
- Frontend flow: The React app initializes. If a user has a valid JWT in local storage, they access the Dashboard. They interact with UI components which trigger Axios requests or Socket.io events.
- Backend flow: Express server listens for HTTP requests and Socket.io events. Middleware intercepts secure API calls to verify JWTs before passing requests to controllers.
- API communication: RESTful architecture where the client requests data (e.g., GET /api/meetings) and the server responds with JSON.
- Database interaction: Controllers use Mongoose models to query or mutate documents in MongoDB (e.g., saving a new user or meeting record).
- Socket connection flow: When a user enters a room, the frontend Socket.io client emits a "join-room" event. The server places the socket in a specific room and broadcasts the presence to other users.
- WebRTC peer connection flow: 
  1. User A joins and signals presence.
  2. User B joins, receives User A's presence, and creates an RTCPeerConnection.
  3. User B creates an 'Offer' and sends it via Socket.io to User A.
  4. User A receives the 'Offer', sets remote description, creates an 'Answer', and sends it back.
  5. Both exchange ICE candidates (network routing info) to establish a direct P2P media stream.
- Authentication flow: User submits credentials -> Server hashes/verifies password -> Server issues JWT -> Client stores JWT -> Client sends JWT in Authorization header for subsequent requests.

--------------------------------------------------------------------------------
5. HOW THE PROJECT WORKS STEP BY STEP
--------------------------------------------------------------------------------
1. User creates account: The user fills out the registration form. The client sends data to /api/auth/register.
2. Password gets encrypted: The backend uses bcrypt to hash the password before saving the new user document to MongoDB.
3. User logs in: User enters email and password. Backend verifies credentials and returns a JWT.
4. JWT token generated: The client stores the JWT and redirects to the Dashboard.
5. User enters dashboard: The client fetches the user's past meetings using the JWT. The user can view history or request AI summaries.
6. User creates/joins room: User clicks "New Meeting" (generates UUID and DB record) or enters a Room ID to join.
7. Socket connection established: Upon entering the Meeting Room component, socket.io-client connects to the server and joins the specific room ID channel.
8. WebRTC exchange starts: The browser requests camera/mic permissions. Once granted, it initiates the Offer/Answer signaling process over the Socket connection.
9. Audio/video stream shared: RTCPeerConnection resolves ICE candidates and establishes a direct connection. Tracks are added to the HTML <video> elements.
10. Real-time communication happens: Users can now see/hear each other, toggle media states, and send text messages via Socket events.

7. IMPORTANT MODULES EXPLAINED
--------------------------------------------------------------------------------
- Authentication module: Consists of authRoutes.js and authController.js. It securely manages user registration and login, ensuring passwords are never stored in plaintext and issuing JWTs for secure access.
- Meeting module: Consists of meetingRoutes.js and meetingController.js. It handles the CRUD operations for meeting sessions, including creating a room, joining a room, and generating AI summaries via OpenAI integration.
- Socket module: Located in sockets/meetingSocket.js. It acts as the traffic controller for real-time events. It manages which users are in which rooms and securely routes WebRTC signaling data (offers, answers, ICE candidates) only to intended recipients.
- Video communication module: The frontend MeetingRoom.jsx and webrtc.js. It leverages the browser's native WebRTC API to acquire local media, establish peer connections, and render incoming remote streams.
- Database module: Centralized in config/db.js and models/. It defines the schemas for Users and Meetings, ensuring data integrity and providing a clean interface for controllers to interact with MongoDB.
- UI module: The collection of React components (e.g., VideoPlayer.jsx, Dashboard.jsx). It translates application state into a responsive, user-friendly interface.

9. DATABASE DESIGN
--------------------------------------------------------------------------------
User schema:
- Fields: name (String), email (String, unique), password (String, hashed).
- Purpose: Stores registered user credentials and profile data.

Meeting schema:
- Fields: title (String), roomId (String, unique UUID), createdBy (ObjectId, ref: User), participants (Array of ObjectIds, ref: User), transcript (String), summary (String), actionItems (Array of objects).
- Purpose: Tracks meeting sessions, who participated, and stores AI-generated summaries.

Relationships:
- A Meeting references a User in createdBy.
- A Meeting references multiple Users in the participants array.

Why MongoDB is suitable:
MongoDB's document-based nature is highly flexible. It allows for easy storage of dynamic arrays like participants and nested objects like actionItems without complex SQL join tables. It perfectly complements the JavaScript ecosystem (JSON).

10. API ENDPOINTS
--------------------------------------------------------------------------------
1. Register API
   - Route: POST /api/auth/register
   - Purpose: Create a new user account.
   - Request body: { "name": "John", "email": "john@ex.com", "password": "pass" }
   - Response: User object and JWT token.

2. Login API
   - Route: POST /api/auth/login
   - Purpose: Authenticate an existing user.
   - Request body: { "email": "john@ex.com", "password": "pass" }
   - Response: User object and JWT token.

3. Create meeting API
   - Route: POST /api/meetings/ (Protected)
   - Purpose: Generate a new meeting room in the database.
   - Request body: { "title": "Optional Title" }
   - Response: Meeting object including the generated roomId.

4. Join meeting API
   - Route: POST /api/meetings/join/:roomId (Protected)
   - Purpose: Add the current user to the participants list of a meeting.
   - Request body: None
   - Response: Meeting object.

5. Generate Summary API
   - Route: POST /api/meetings/:roomId/generate-summary (Protected)
   - Purpose: Use AI to process a transcript and return actionable insights.
   - Request body: { "transcript": "Meeting text..." }
   - Response: Generated summary and action items.

11. SECURITY IMPLEMENTATION
--------------------------------------------------------------------------------
- Password hashing: Implemented using bcryptjs with a salt round of 10. Passwords are never logged or stored in plain text.
- JWT verification: A robust custom middleware checks the Authorization header for a valid Bearer token. If invalid or expired, the request is rejected with a 401 Unauthorized status.
- Protected routes: React Router wraps restricted pages in a <ProtectedRoute> component, redirecting unauthenticated users to the login screen. Backend routes use the protect middleware to secure API data.
- Secure API handling: CORS (Cross-Origin Resource Sharing) is configured to restrict unauthorized domains from accessing the API.
- Environment variables: Sensitive data like JWT_SECRET, MONGO_URI, and OPENAI_API_KEY are stored in .env files and never committed to version control.

--------------------------------------------------------------------------------
12. CHALLENGES FACED IN THIS PROJECT
--------------------------------------------------------------------------------
- WebRTC connection issues: Managing the asynchronous nature of WebRTC signaling. Ensuring ICE candidates are queued if the remote description isn't set yet was a significant hurdle.
- Socket synchronization: Handling edge cases where a user disconnects abruptly or refreshing the page breaks the socket connection state.
- Media device permissions: Browsers block autoplaying video and strictly enforce HTTPS or localhost for accessing the camera and microphone, requiring careful error handling for denied permissions.
- State management: Managing complex, interconnected states in the React component (e.g., local stream, remote stream, UI toggles) without causing infinite re-renders.
- AI API constraints: Handling OpenAI rate limits and designing fallback mechanisms (mock summaries) to ensure the application remains functional even when API quotas are exceeded.

15. COMPLETE PROJECT WORKFLOW SUMMARY
--------------------------------------------------------------------------------
1. Frontend: The user accesses the React app, authenticates, and requests to join a meeting.
2. Backend: Express validates the JWT, and MongoDB logs the user as a participant in the requested meeting room.
3. Socket.io: The server registers the user in a specific Socket room and alerts existing members.
4. WebRTC: The browser acquires the camera/mic. The frontend uses Socket.io to securely exchange WebRTC Offers, Answers, and ICE candidates with the remote peer.
5. User Communication: A direct, secure Peer-to-Peer connection is established. Video, audio, and data (chat messages) flow seamlessly in real-time. Post-meeting, the user can utilize the AI service for an intelligent summary.
